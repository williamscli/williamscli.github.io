Resolved

The same mkdir works in your terminal but fails in Claude Code.

If your project lives in a synced OneDrive folder, Claude Code's Bash tool dies on a plain directory create with a cryptic COM error. The cause isn't Claude Code. The fix is one line.

claude-code — Bash tool
● Bash(mkdir "C:\Users\...\OneDrive - Contoso\Desktop\test")
  ⎿  Error: Exit code 1
     Class not registered

01 — The symptom

One command, two outcomes.

You run mkdir against a OneDrive folder in your own terminal and it works. Claude Code runs the exact same command and fails with Class not registered. That asymmetry is the whole answer. This is not a Claude Code bug. It is two different programs, both named mkdir.

  • Your terminal uses the native Windows mkdir.
  • Claude Code's Bash tool shells out to Git Bash, whose mkdir comes from MSYS2 — a POSIX emulation layer.

Same command, two implementations. Native Windows handles the OneDrive folder. The POSIX one walks into a wall.

02 — Root cause

OneDrive folders aren't quite folders.

With Files On-Demand on, OneDrive stores synced folders as reparse points — cloud placeholders Windows resolves through a sync-provider COM object the first time a program touches them. Git Bash's MSYS2 mkdir, running as a spawned child process, lands where that COM class can't be instantiated. Windows returns 0x80040154 — REGDB_E_CLASSNOTREG — surfaced as the vague "Class not registered."

Same command, two shells. Only the native path resolves the OneDrive placeholder.

The asymmetry isn't Claude Code versus you. It's Git Bash's mkdir versus native Windows mkdir — one knows how to wake a cloud placeholder, the other doesn't.

03 — Why it matters

It's not just mkdir. It's your Office files.

Real edits to .xlsx and .docx run through the shell. They're binary formats: when Claude changes a cell or document structure, it writes and runs a Python script — openpyxl, python-docx — through the Bash tool. If Bash can't operate in your OneDrive folder, that whole workflow is dead. Fix the shell and you unlock editing Office documents in place, where they already sync.

04 — The fix

Switch Claude Code onto the native PowerShell tool.

PowerShell is a native Windows shell, so its file operations resolve OneDrive placeholders the same way your terminal does. The switch is one flag: CLAUDE_CODE_USE_POWERSHELL_TOOL=1. You need Claude Code v2.1.84 or newer.

  1. Close Claude Code

    Settings are read only on a fresh start.

  2. Check for an existing settings file

    Run Test-Path "$env:USERPROFILE\.claude\settings.json". False, create it. True, merge into it so you don't clobber existing keys.

  3. Add the flag inside an env block

    Set "CLAUDE_CODE_USE_POWERSHELL_TOOL": "1", nested under "env" in the JSON.

  4. Validate the JSON

    A stray comma makes Claude Code ignore the file silently. Confirm it parses with ConvertFrom-Json.

  5. Restart and confirm

    Retry the mkdir that failed. No error means it took, and in-place .xlsx editing on OneDrive now works.

Alt

Prefer a system-wide flag? Run [Environment]::SetEnvironmentVariable("CLAUDE_CODE_USE_POWERSHELL_TOOL", "1", "User"), then open a brand-new terminal so it's inherited.

05 — Belt and suspenders

Two supporting moves.

  • Hydrate the folder. Right-click it in Explorer, choose "Always keep on this device." Cloud-only placeholders are the fragile case.
  • Or sidestep OneDrive. Work in a plain local folder like C:\dev\ and copy finished files into OneDrive. No reparse points, no COM, no surprises.
Note

The PowerShell tool is a preview feature. It doesn't cover a couple of edge modes — auto-mode, Windows sandboxing — none of which affect editing Office files on OneDrive.
